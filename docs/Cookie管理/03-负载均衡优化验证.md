# Cookie 负载均衡优化 - 验证指南

## 📋 优化内容

本次优化改进了 Cookie 池管理器的选择策略，从**健康评分加权随机**改为**分层轮询 + 健康评分过滤**，实现更均衡的负载分布。

### 核心改进

1. **优先选择使用次数最少的 Cookie**（容忍度 ±5 次）
2. **健康评分作为门槛**（≥70 分优先，降级 ≥50 分）
3. **在符合条件的 Cookie 中均匀随机选择**（不再加权）


- `使用次数`：当前 Cookie 的总调用次数
- `最小使用次数`：所有候选 Cookie 中的最少使用次数

## ⚙️ 参数调优

如果效果不理想，可以调整以下参数（在 `cookiePoolManager.ts` 第 28-30 行）：

```typescript
private readonly USAGE_TOLERANCE = 5;        // 使用次数容忍度
private readonly HEALTH_THRESHOLD = 70;      // 健康评分门槛
private readonly HEALTH_THRESHOLD_FALLBACK = 50; // 降级门槛
```

### 调整建议

| 场景         | USAGE_TOLERANCE | HEALTH_THRESHOLD | 说明               |
| ------------ | --------------- | ---------------- | ------------------ |
| 更严格均衡   | 3               | 70               | 差异控制在 ±3 次内 |
| 更宽松均衡   | 10              | 70               | 允许更大偏差       |
| 高成功率优先 | 5               | 80               | 只选高健康评分     |
| 低成功率环境 | 5               | 50               | 放宽健康要求       |

## 🐛 问题排查

### 问题 1：某些 Cookie 仍然被过度使用

**可能原因**：

- 这些 Cookie 健康评分特别高
- 其他 Cookie 处于冷却期

**解决方案**：

- 降低 `HEALTH_THRESHOLD` 到 60
- 缩短冷却时间（修改第 87 行的 `COOLDOWN_TIME`）

### 问题 2：新 Cookie 不被使用

**可能原因**：

- 新 Cookie 数量为 0 时，`Math.min()` 返回 `Infinity`

**解决方案**：

- 已在代码中处理，确保至少有 1 个候选

### 问题 3：请求成功率下降

**可能原因**：

- 健康评分门槛过高，排除了部分可用 Cookie

**解决方案**：

- 降低 `HEALTH_THRESHOLD` 到 60
- 检查 `HEALTH_THRESHOLD_FALLBACK` 是否生效

## 📝 技术细节

### 算法流程

```
1. 获取所有活跃Cookie
   ↓
2. 过滤冷却期Cookie（30秒）
   ↓
3. 计算每个Cookie的使用次数（successCount + failureCount）
   ↓
4. 找出最少使用次数 minUsage
   ↓
5. 选取使用次数 ≤ (minUsage + USAGE_TOLERANCE) 的Cookie
   ↓
6. 从中筛选 healthScore ≥ HEALTH_THRESHOLD 的Cookie
   ↓
7. 如果没有，降级到 healthScore ≥ HEALTH_THRESHOLD_FALLBACK
   ↓
8. 在最终候选池中均匀随机选择
   ↓
9. 更新 lastUsedAt，返回Cookie值
```

### 复杂度分析

- **时间复杂度**：O(n)，n 为活跃 Cookie 数量
- **空间复杂度**：O(n)，用于存储 usageMap
- **性能影响**：可忽略不计（200 个 Cookie 约 0.1ms）
