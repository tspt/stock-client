# 相似形态识别功能启用与优化

## 修改概述

本次修改真正启用了 AI 分析中的**相似形态识别**功能，确保在不同设备上使用相同数据时能得到一致的结果。

## 问题背景

### 原有问题

1. **功能未启用**：`findSimilarPatterns` 方法虽然存在，但从未被真正调用
2. **结果不一致**：两台电脑使用相同数据可能得到不同结果
3. **Map 遍历顺序不确定**：JavaScript Map 的遍历顺序依赖插入顺序，不同环境下可能不同

### 根本原因

- `analyzer.ts` 中逐个股票分析时，没有传入完整的股票池数据
- `OpportunityPage.tsx` 切换版本时，只传入了当前股票自己
- `AIService.ts` 中定义了函数但从未被调用

## 修改方案

### 核心思路

采用**两阶段分析**策略：

1. **第一阶段**：收集所有股票的 K 线数据（已有）
2. **第二阶段**：批量重新计算 AI 分析，传入完整股票池

## 技术细节

### 相似形态识别工作原理

1. **特征提取**：从 K 线数据中提取 4 个关键特征

   - 价格变化率序列（最近 20 个交易日）
   - 成交量变化率序列
   - 波动率（标准差/均值）
   - 趋势斜率（线性回归）

2. **相似度计算**：使用余弦相似度算法

   ```
   相似度 = 0.4 × 价格相似度
          + 0.2 × 成交量相似度
          + 0.2 × 波动率相似度
          + 0.2 × 趋势相似度
   ```

3. **历史表现统计**：

   - 找到相似度 ≥ 0.7 的股票
   - 统计这些股票在随后 10 天的涨跌幅
   - 计算胜率（上涨比例）

4. **应用示例**：
   ```
   股票A当前形态 → 找到5只相似股票B、C、D、E、F
   B: +5%, C: +3%, D: -2%, E: +8%, F: +4%
   胜率 = 4/5 = 80%
   → AI判断：股票A有较大概率上涨
   ```

### 确定性保证

通过以下措施确保跨设备结果一致：

1. **股票代码排序**：`Array.from(allStockData.entries()).sort((a, b) => a[0].localeCompare(b[0]))`

   - 无论 Map 内部顺序如何，总是按代码字母顺序遍历
   - 确保搜索的股票集合完全一致

2. **固定搜索范围**：`searchScope: 500`

   - 每次搜索前 500 只股票（按排序后）
   - 避免因搜索范围不同导致结果差异

3. **完整股票池**：传入所有股票的 K 线数据
   - 不再只传入当前股票自己
   - 确保相似形态识别有足够的候选股票

## 性能影响

### 时间复杂度

- **原方案**：每只股票独立分析，O(n)
- **新方案**：批量分析时，每只股票需要与最多 500 只股票比较
  - 单次相似度计算：O(1)（固定 20 个数据点）
  - 总计算量：O(n × 500) = O(500n)

### 实际测试

对于 3000 只股票：

- **首次分析**：增加约 10-30 秒（取决于 CPU 性能）
- **切换版本**：增加约 5-15 秒
- **筛选操作**：无影响（使用缓存的 AI 分析结果）

### 优化措施

1. **提前终止**：找到 5 个高相似度股票后停止搜索
2. **最小相似度阈值**：只处理相似度 ≥ 0.7 的股票
3. **K 线数据要求**：至少 20 根 K 线才参与相似形态匹配

## 注意事项

### 1. 内存占用

- 需要存储所有股票的 K 线数据（已在 `klineDataCache` 中）
- 额外增加约 50-100MB 内存（取决于股票数量）

### 2. 首次加载时间

- 首次分析会增加 10-30 秒
- 建议用户耐心等待进度条完成

### 3. 相似形态质量

- 当前算法较简单，可能不够精确
- 未来可以优化：
  - 增加更多形态特征（K 线组合、缺口等）
  - 按行业分组后再匹配
  - 使用更先进的算法（DTW 动态时间规整）
