## 🔧 技术改进详情

### 1. 动态加载实现

**改进前的问题**:

```javascript
// ❌ 硬编码方式
const stockFiles = [
  { file: '三孚股份.json', dates: ['2025-12-22', '2026-04-01'] },
  { file: '三维通信.json', dates: ['2025-08-22', '2026-01-07', '2026-04-24'] },
  // ... 需要手动维护 69 只股票
];
```

**改进后的实现**:

```javascript
// ✅ 自动扫描方式
const STOCK_DATA_DIR = path.join(__dirname, '..', '股票数据');

function loadStockFiles() {
  const files = fs.readdirSync(STOCK_DATA_DIR).filter((f) => f.endsWith('.json'));

  const stockFiles = files.map((filename) => {
    const filePath = path.join(STOCK_DATA_DIR, filename);
    const content = fs.readFileSync(filePath, 'utf-8');
    const jsonData = JSON.parse(content);

    return {
      file: filename,
      dates: jsonData.buypointDate || [], // 从文件自动读取
    };
  });

  return stockFiles;
}

const stockFiles = loadStockFiles();
```

**核心优势**:

- 零维护: 添加新股票只需放入文件，无需修改代码
- 自动同步: 始终与股票数据目录保持一致
- 减少错误: 直接从文件读取，避免手动输入错误

## Bug 修复记录

### Bug: 未捕获买点数显示错误

**问题现象**:

```
Config_1: 召回率 12.2%, 未捕获买点: 188个 / 188个  ← ❌ 始终显示188
Config_22: 召回率 87.2%, 未捕获买点: 188个 / 188个  ← ❌ 应该显示24
```

**根本原因**:

```javascript
// ❌ 错误代码
const missedBuypoints = positiveSamples; // 所有正样本，而非未捕获的
```

**修复方案**:

```javascript
// ✅ 根据预测结果计算真正的 FN（False Negative）
const missedCount = metrics.fn;
const missedIndices = [];
for (let i = 0; i < y.length; i++) {
  if (y[i] === 1 && predictions[i] === 0) {
    missedIndices.push(i);
  }
}

// 提取真正的未捕获买点
const missedSamples = missedIndices.map((idx) => positiveSamples[idx]);
```

**修复效果**:

```
Config_1: 召回率 12.2%, 未捕获买点: 165个 / 188个  ← ✅ 正确
Config_22: 召回率 87.2%, 未捕获买点: 24个 / 188个   ← ✅ 正确
```

## 🔬 特征工程探索（已还原）

### 12 特征版本尝试

**新增的 4 个特征**:

1. **RSI(14)** - 相对强弱指标

   - 作用: 捕捉超买超卖状态
   - 预期: 识别 RSI 中性区域（40-60）的买点

2. **volChangeRate** - 成交量变化率

   - 公式: `(今日量 - 5日均量) / 5日均量`
   - 作用: 捕捉放量/缩量

3. **consecutiveUpDays** - 连续上涨天数

   - 作用: 捕捉温和上涨模式（2-3 天）

4. **bbWidthPercent** - 布林带宽度百分比
   - 公式: `(上轨 - 下轨) / 中轨 * 100`
   - 作用: 更精确的波动率指标

**训练结果**:

- 训练集召回率: 98.9% ⬆️（8 特征版为 87.2%）
- 真实回测覆盖率: 78.72% ⬆️（8 特征版为 78.19%，仅提升 0.53%）

### 特征工程失败原因分析

**过拟合问题**:

| 版本        | 特征数 | 训练集召回率 | 真实覆盖率 | 过拟合程度     |
| ----------- | ------ | ------------ | ---------- | -------------- |
| v4.0 原版   | 8      | 87.2%        | 78.19%     | 轻微（9.01%）  |
| v4.0 增强版 | 12     | 98.9%        | 78.72%     | 严重（20.18%） |

**结论**:

- 新增的 4 个特征在训练集上表现优异，但在真实回测中泛化能力不足
- 过拟合程度从 9.01% 增加到 20.18%，说明模型学习了过多噪声
- 真实覆盖率仅提升 0.53%，性价比极低

### 最终决策：还原到 8 特征版本

**原因**:

1. ✅ 更简洁，易于维护
2. ✅ 过拟合程度低（9.01% vs 20.18%）
3. ✅ 真实覆盖率相当（78.19% vs 78.72%）
4. ✅ 模型文件更小，训练更快

**还原操作**:

- 移除 `calculateRSI()` 和 `calculateConsecutiveUpDays()` 函数
- 从 `FEATURE_NAMES` 数组中移除 4 个新特征
- 重新训练模型并部署
