## ✅ 已修复的问题

### 1. 数据长度要求不一致

**问题描述：**

- 支撑阻力位检查函数要求 `klineData.length < 20` 就返回 false
- 但 `identifySupportResistance` 函数默认需要 60 天数据（`lookbackPeriod: 60`）
- 导致即使有 20-59 天数据，也会因为内部检查而返回空结果

**修复方案：**

```typescript
// 修改前
if (!klineData || klineData.length < 20) {
  return false;
}

// 修改后
if (!klineData || klineData.length < 60) {
  // 支撑阻力位识别至少需要60天数据
  return false;
}
```

**影响：** 确保前置检查与内部算法要求一致，避免无效计算。

### 2. 成交量异常提醒缺少回退机制

**问题描述：**
原始实现中，成交量异常提醒只有首次触发判断：

```typescript
if (anomalyResult.isAnomaly && !triggered) {
  return true;
}
```

这会导致：

- 如果成交量持续保持高位（如连续几天都是 2 倍以上），只会触发一次
- 但如果成交量回落后再升高，由于 `triggered=true`，无法再次触发
- 缺少回退状态的跟踪

**修复方案：**

```typescript
if (anomalyResult.isAnomaly) {
  if (!triggered) {
    // 首次触发
    return true;
  }
  // 回退机制：成交量必须回落到阈值以下，再异常才再次触发
  if (lastTriggerPrice !== undefined) {
    const historicalVolumes = klineData.slice(-volumePeriod - 1, -1).map((k) => k.volume);
    const avgVolume =
      historicalVolumes.reduce((sum, vol) => sum + vol, 0) / historicalVolumes.length;
    const lastVolumeRatio = lastTriggerPrice / avgVolume;

    // 如果上次触发后成交量已经回落到正常水平，则可以再次触发
    if (lastVolumeRatio < volumeMultiplier) {
      return true;
    }
  }
}
```

同时在回退更新逻辑中添加：

```typescript
else if (alert.type === 'volume_anomaly' && klineData) {
  const currentVolumeRatio = klineData[klineData.length - 1].volume / avgVolume;

  // 如果成交量已经回落到阈值以下，更新 lastTriggerPrice
  if (
    currentVolumeRatio < volumeMultiplier &&
    (lastTriggerPrice === undefined || lastTriggerPrice >= avgVolume * volumeMultiplier)
  ) {
    get().updateAlert(alert.id, {
      lastTriggerPrice: klineData[klineData.length - 1].volume
    });
  }
}
```

**影响：** 现在成交量异常提醒可以正确地多次触发，每次都需要"异常 → 正常 → 异常"的完整周期。

### 3. 技术指标金叉/死叉缺少回退机制

**问题描述：**
原始实现同样只有首次触发判断：

```typescript
if (condition === 'golden_cross' && crossResult.isGoldenCross && !triggered) {
  return true;
}
```

问题：

- MACD/KDJ/RSI 金叉后，如果快线一直在慢线上方，无法再次触发
- 即使出现第二次金叉（快线下穿后再上穿），也不会提醒

**修复方案：**

```typescript
if (condition === 'golden_cross') {
  if (crossResult.isGoldenCross) {
    if (!triggered) {
      // 首次触发
      return true;
    }
    // 回退机制：快线必须回落至慢线下方，再金叉才再次触发
    if (lastTriggerPrice !== undefined && crossResult.fastValue <= crossResult.slowValue) {
      return true;
    }
  }
}
```

回退状态更新：

```typescript
else if (alert.type === 'indicator_cross' && klineData) {
  const crossResult = detectIndicatorCross(klineData, indicatorType!, maFastPeriod, maSlowPeriod);

  if (condition === 'golden_cross') {
    // 金叉提醒：如果快线回落至慢线下方，更新 lastTriggerPrice
    if (
      crossResult.fastValue <= crossResult.slowValue &&
      (lastTriggerPrice === undefined || lastTriggerPrice === 1)
    ) {
      get().updateAlert(alert.id, { lastTriggerPrice: 0 });
    }
  } else if (condition === 'death_cross') {
    // 死叉提醒：如果快线回升至慢线上方，更新 lastTriggerPrice
    if (
      crossResult.fastValue >= crossResult.slowValue &&
      (lastTriggerPrice === undefined || lastTriggerPrice === 0)
    ) {
      get().updateAlert(alert.id, { lastTriggerPrice: 1 });
    }
  }
}
```

**说明：**

- 使用 `lastTriggerPrice` 字段存储状态标志（0=未交叉/已回退，1=已触发）
- 金叉：需要快线先下穿慢线（回退），再上穿才能再次触发
- 死叉：需要快线先上穿慢线（回退），再下穿才能再次触发

**影响：** 指标交叉提醒现在可以正确检测多次交叉事件。

### 4. 函数参数顺序统一

**问题描述：**
新增的三个检查函数参数顺序不统一：

- `shouldTriggerSupportResistanceAlert(alert, currentPrice, klineData)` ✅
- `shouldTriggerVolumeAnomalyAlert(alert, klineData)` ❌ 缺少 currentPrice
- `shouldTriggerIndicatorCrossAlert(alert, klineData)` ❌ 缺少 currentPrice

**修复方案：**
统一所有函数签名为：

```typescript
function shouldTriggerXXXAlert(
  alert: PriceAlert,
  currentPrice: number, // 统一添加
  klineData?: KLineData[]
): boolean;
```

并更新所有调用点。

**影响：** 提高代码一致性，便于后续维护和扩展。

## ⚠️ 已知限制和建议

### 1. K 线数据依赖

**现状：**

- 新增的三种提醒类型需要 K 线数据
- 目前仅在有机会分析数据的股票上生效（因为那里有 K 线数据缓存）

**建议：**

- 短期：在用户设置提醒时提示需要机会分析数据
- 长期：为普通自选股也自动获取并缓存 K 线数据

### 2. 性能考虑

**现状：**

- 支撑阻力位识别每次都会重新计算（局部极值 + 聚类）
- 如果有大量此类提醒，可能影响性能

**建议：**

- 实现支撑阻力位缓存机制（例如每小时更新一次）
- 或者仅在价格接近关键位时才重新计算

### 3. 参数调优

**现状：**

- 支撑阻力位：60 天回溯，5 天窗口，2%容差
- 成交量异常：20 天周期，2 倍阈值
- 这些是通用参数，可能不适合所有股票

**建议：**

- 提供参数自定义选项
- 或基于股票特性（如波动率、市值）自动推荐参数

### 4. 误报控制

**现状：**

- 所有提醒都有回退机制，但仍可能出现频繁触发

**建议：**

- 添加冷却时间（例如同一提醒 24 小时内最多触发一次）
- 或添加触发频率统计，让用户了解提醒质量
